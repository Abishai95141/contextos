## 1. Component Architecture

### System Overview

ContextOS is a cloud-hosted platform. Nothing runs on the developer's machine except the VS Code extension (a thin client) and Claude Code's hook configuration (auto-generated curl commands). All state — Feature Packs, Context Packs, policy rules, run history — lives in managed services (Supabase PostgreSQL and Upstash Redis).

### Agent-Specific Configuration

ContextOS integrates with three AI agents. Each has different configuration formats:

#### Claude Code — Hooks + MCP

**Hooks** (`.claude/settings.json`):
- SessionStart: `type: "command"` only (the only type SessionStart supports)
- PreToolUse, PostToolUse: `type: "http"` (native HTTP hook support)
- SessionEnd: `type: "http"` (fires once at session termination)
- MCP tools appear as `mcp__contextos__<tool>` in hook events

**MCP** (`.mcp.json` or `claude mcp add --transport http`):
```json
{
  "mcpServers": {
    "contextos": {
      "type": "http",
      "url": "https://mcp.contextos.dev/mcp",
      "headers": {
        "Authorization": "Bearer $CONTEXTOS_TOKEN"
      }
    }
  }
}
```

#### VS Code + Copilot — MCP Only

Config file: `.vscode/mcp.json` (note: key is `"servers"`, not `"mcpServers"`)
```json
{
  "servers": {
    "contextos": {
      "type": "http",
      "url": "https://mcp.contextos.dev/mcp",
      "headers": {
        "Authorization": "Bearer ${env:CONTEXTOS_TOKEN}"
      }
    }
  }
}
```

> VS Code Copilot has no hooks system — it relies solely on MCP tools for context injection.

#### Cursor — MCP Only

Config file: `.cursor/mcp.json` (key is `"mcpServers"`)
```json
{
  "mcpServers": {
    "contextos": {
      "url": "https://mcp.contextos.dev/mcp",
      "headers": {
        "Authorization": "Bearer ${env:CONTEXTOS_TOKEN}"
      }
    }
  }
}
```

> Cursor auto-detects Streamable HTTP vs SSE from the URL. Supports `${env:NAME}` interpolation for secrets.

#### Summary: What Each Agent Gets

| Capability | Claude Code | VS Code + Copilot | Cursor |
|-----------|-------------|-------------------|--------|
| MCP tools (get_feature_pack, etc.) | ✅ via MCP | ✅ via MCP | ✅ via MCP |
| Hook-based policy enforcement | ✅ via hooks | ❌ | ❌ |
| Automatic context injection | ✅ via SessionStart hook | ❌ (use MCP `get_feature_pack` manually) | ❌ (use MCP `get_feature_pack` manually) |
| Run recording & context packs | ✅ via hooks pipeline | Partial (via MCP `save_context_pack`) | Partial (via MCP `save_context_pack`) |

The VS Code extension auto-generates the correct config file format for all three agents.

## 2. Request Flows — All 4 Hook Events

These are the exact step-by-step flows that occur when Claude Code fires each hook. Every step is numbered. Implementation must follow this sequence.

### Hook 1: SessionStart

**Trigger:** Developer opens Claude Code in a project directory.

> **Hook type:** SessionStart only supports `type: "command"` hooks in Claude Code — not HTTP. The VS Code extension generates a curl command that pipes stdin to our hooks-bridge HTTP endpoint.

```
Developer opens Claude Code in /home/dev/myapp
     │
     │  Claude Code fires command hook (curl posts stdin JSON to hooks-bridge)
     ▼
POST https://hooks.contextos.dev/hooks/session-start
  Headers: Authorization: Bearer $CONTEXTOS_TOKEN
           Content-Type: application/json
  Body: {
    "session_id": "abc123-session-uuid",
    "cwd": "/home/dev/myapp",
    "transcript_path": "/home/dev/.claude/projects/myapp/transcript.jsonl",
    "hook_event_name": "SessionStart",
    "source": "startup",
    "model": "claude-sonnet-4-6"
  }
     │
     ▼
hooks-bridge receives request
```

**Step-by-step execution inside hooks-bridge:**

1. **Validate token** — Call `verifyToken(token, { secretKey: CLERK_SECRET_KEY })` via `@clerk/backend`. Extract `orgId` and `userId` from claims.
2. **Parse payload** — Validate request body against `HookPayloadSchema` (Zod). If invalid → respond `200`, log warning.
3. **Look up project** — Query `SELECT id, slug FROM projects WHERE clerk_org_id = $orgId` and match `cwd` against known project paths (stored in project metadata or matched by repository root convention).
4. **No project match?** → Respond `200 { continue: true }` immediately. Log warning: `{ sessionId, cwd, orgId } — "No matching project for cwd"`. Non-blocking — Claude Code proceeds normally.
5. **Create run record** — `INSERT INTO runs (project_id, session_id, idempotency_key, status, cwd) VALUES ($projectId, $sessionId, 'run:' || $projectId || ':' || $sessionId, 'in_progress', $cwd) ON CONFLICT (idempotency_key) DO NOTHING RETURNING id`. If conflict → existing run, fetch its `id`.
6. **Cache run ID in Redis** — `SET session:$sessionId:run_id $runId EX 86400` (24-hour TTL).
7. **Log** — `{ sessionId, runId, cwd, projectSlug } — "Session started"`.
8. **Respond** — `200` with optional `additionalContext` containing project context. For SessionStart, stdout text or `hookSpecificOutput.additionalContext` is added to Claude's context.

**Total latency target:** < 50ms (DB insert + Redis write are fast; no policy evaluation needed).

---

### Hook 2: PreToolUse

**Trigger:** Claude Code decides to use a tool (e.g., `Bash`, `Edit`, `Write`). This hook is **synchronous** — the tool does not execute until the hook responds.

```
Claude Code wants to run: Bash { command: "rm -rf dist" }
     │
     │  HTTP POST (blocks until response)
     ▼
POST https://hooks.contextos.dev/hooks/pre-tool-use
  Body: {
    "session_id": "abc123-session-uuid",
    "hook_event_name": "PreToolUse",
    "tool_name": "Bash",
    "tool_input": { "command": "rm -rf dist" },
    "tool_use_id": "tool-use-uuid-456"
  }
     │
     ▼
hooks-bridge receives request
```

**Step-by-step execution:**

1. **Validate token** — Same as SessionStart.
2. **Parse payload** — Validate against `HookPayloadSchema`.
3. **Read run ID from Redis** — `GET session:$sessionId:run_id`. If not found → default allow (session not tracked), log warning.
4. **Load policy rules** — Check Redis cache first: `GET policy:project:$projectId:rules`.
   - **Cache hit:** Use cached rules (JSON array).
   - **Cache miss:** Query `SELECT * FROM policy_rules pr JOIN policies p ON pr.policy_id = p.id WHERE p.project_id = $projectId AND p.is_active = true AND pr.is_active = true ORDER BY pr.priority ASC`. Cache in Redis: `SET policy:project:$projectId:rules $json EX 60`.
5. **Evaluate rules (Policy Engine)** — For each rule in priority order:
   - a. If `rule.event_type != 'PreToolUse' AND rule.event_type != '*'` → skip.
   - b. If `rule.tool_pattern` defined → glob-match against `tool_name` using `micromatch`. No match → skip.
   - c. If `rule.path_pattern` defined → extract `file_path` from `tool_input` (check `tool_input.file_path`, `tool_input.path`, `tool_input.command` for path extraction). No path extractable → skip path check. No glob match → skip.
   - d. **MATCH** → return `{ decision: rule.decision, ruleId: rule.id, reason: rule.reason }`.
6. **No rule matched** → `{ decision: 'allow', ruleId: null, reason: 'default allow — no matching rule' }`.
7. **Async: record event** — Enqueue BullMQ job to `record-run-event` queue: `{ runId, eventType: 'policy_check', toolName, toolInput, policyDecision, idempotencyKey: $sessionId:$toolUseId:pre }`. This happens AFTER the response is sent. Do not wait for it.
8. **Async: record policy decision** — Insert `policy_decisions` row with idempotency key `pd:$sessionId:$toolName:PreToolUse`. `ON CONFLICT DO NOTHING`.
9. **Respond based on decision:**
   - `deny` → `200 { hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "deny", permissionDecisionReason: "$ruleName: $reason" } }`
   - `warn` → `200 { hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "allow" } }` + async log the warning.
   - `allow` → `200 { hookSpecificOutput: { hookEventName: "PreToolUse", permissionDecision: "allow" } }`

**Total latency target:** < 200ms. Redis read (~2ms) + in-memory rule evaluation (~1ms) + response. DB writes are async.

**Timeout behavior:** If policy evaluation exceeds 500ms → default to `allow`, log the timeout. Never block Claude Code.

### Hook 3: PostToolUse

**Trigger:** Tool execution completed. Claude Code sends the result. This is non-blocking — Claude Code does not wait.

```
Tool completed: Bash returned { success: true, output: "removed dist/" }
     │
     │  HTTP POST (non-blocking)
     ▼
POST https://hooks.contextos.dev/hooks/post-tool-use
  Body: {
    "session_id": "abc123-session-uuid",
    "hook_event_name": "PostToolUse",
    "tool_name": "Bash",
    "tool_input": { "command": "rm -rf dist" },
    "tool_response": { "success": true, "output": "removed dist/" },
    "tool_use_id": "tool-use-uuid-456"
  }
     │
     ▼
hooks-bridge receives request
```

**Step-by-step execution:**

1. **Validate token**.
2. **Parse payload**.
3. **Read run ID from Redis** — `GET session:$sessionId:run_id`.
4. **Enqueue event recording** — BullMQ job to `record-run-event` queue: `{ runId, eventType: 'tool_use', toolName, toolInput, toolOutput: tool_response, idempotencyKey: $sessionId:$toolUseId:post }`.
5. **Respond immediately** — `200 { continue: true }`. Do not wait for DB write.
6. **Log** — `{ toolName, success, sessionId, runId } — "Tool use completed"`.

**Total latency target:** < 10ms (validate + enqueue + respond).

---

### Hook 4: SessionEnd

**Trigger:** Claude Code session terminates (user exits, `/clear`, session switch via `/resume`, or logout). This is the cleanup hook — it triggers run finalization and context pack assembly.

> **Why SessionEnd, not Stop?** Claude Code's `Stop` event fires every time Claude finishes responding (i.e., after each turn), not when the session ends. `SessionEnd` fires exactly once when the session actually terminates — matching our run-finalization semantics. See [hooks reference](https://code.claude.com/docs/en/hooks) for the full lifecycle.

```
Session terminates (user exits, /clear, /resume switch, logout)
     │
     │  HTTP POST (non-blocking, 1.5s default timeout)
     ▼
POST https://hooks.contextos.dev/hooks/session-end
  Body: {
    "session_id": "abc123-session-uuid",
    "hook_event_name": "SessionEnd",
    "cwd": "/home/dev/myapp",
    "reason": "other"
  }
     │
     ▼
hooks-bridge receives request
```

**Step-by-step execution:**

1. **Validate token**.
2. **Parse payload**.
3. **Read run ID from Redis** — `GET session:$sessionId:run_id`. If not found → respond `200`, log warning (untracked session).
4. **Update run status** — `UPDATE runs SET status = 'completed', completed_at = now() WHERE id = $runId`.
5. **Enqueue context pack assembly** — BullMQ job to `context-pack-assembly` queue:
   ```json
   {
     "runId": "...",
     "sessionId": "...",
     "projectId": "...",
     "cwd": "/home/dev/myapp"
   }
   ```
   Job options: `{ attempts: 3, backoff: { type: "exponential", delay: 5000 } }`.
6. **Clean up Redis** — `DEL session:$sessionId:run_id`.
7. **Log** — `{ sessionId, runId, durationMs } — "Session ended, context pack assembly enqueued"`.
8. **Respond** — `200` (empty body or `{}`). SessionEnd has no decision control — it cannot block termination.

**Timeout note:** SessionEnd has a 1.5s default timeout (configurable via `CLAUDE_CODE_SESSIONEND_HOOKS_TIMEOUT_MS`). Our handler targets <50ms (DB update + Redis ops + enqueue), well within budget.

**Context pack assembly worker (async, runs after response is sent):**

1. Load all `run_events` for the run:
   `SELECT * FROM run_events WHERE run_id = $runId ORDER BY created_at ASC`.
2. Extract file paths from `Edit`/`Write` tool events.
3. Generate raw git diff (from `cwd`, if accessible from container — otherwise use file paths from events).
4. Call semantic-diff service:
   `POST $SEMANTIC_DIFF_URL/analyze` with `{ run_id, raw_diff, feature_pack_context }`.
   Response: `{ summary, apis_added, apis_removed, tests_added, tests_broken, new_modules, key_decisions }`.
5. Assemble context pack content in markdown format:
   ```
   # Context Pack: [Summary from semantic-diff]

   ## Session
   - Run ID: {runId}
   - Started: {startedAt} | Completed: {completedAt}
   - Feature Pack: {featurePackName} v{version}

   ## Semantic Diff Summary
   {summary from semantic-diff service}

   ## APIs Added
   {list}

   ## Files Modified
   {extracted from events}

   ## Tool Use Log
   {summarized from run_events}

   ## Key Decisions
   {from semantic-diff + record_decision events}
   ```
6. Insert `context_packs` row: `INSERT INTO context_packs (project_id, run_id, content, semantic_diff, summary, status) VALUES (...)`.
7. Enqueue embedding job: BullMQ job to `embed-context-pack` queue with `{ contextPackId, content, projectId }`.

## 3. MCP Server — Tool and Resource Flows

The MCP server speaks MCP protocol over Streamable HTTP. Agents (Claude Code, Cursor, Copilot) connect as MCP clients.

### Connection Flow

```
Agent (MCP client)
     │
     │  HTTP POST https://mcp.contextos.dev/mcp
     │  Headers: Authorization: Bearer <clerk-m2m-token>
     │  Body: MCP JSON-RPC request
     ▼
Express middleware:
  1. verifyToken() via @clerk/backend → extract orgId, userId
  2. Attach { orgId, userId } to request context
     │
     ▼
StreamableHTTPServerTransport (@modelcontextprotocol/sdk)
  Parses MCP JSON-RPC → routes to tool/resource handler
     │
     ▼
Tool handler executes:
  - All DB queries scoped to orgId (never leaks cross-org data)
  - Returns { content: [{ type: 'text', text: '...' }] }
  - On error: returns { content: [...], isError: true }
  - NEVER throws from a tool handler
```

### Tool: get_feature_pack — Inheritance Resolution

This is the most complex tool. It resolves Feature Pack inheritance (child overrides parent).

```
Input: { projectSlug: "my-app", packSlug: "auth-module" }
     │
     ▼
1. SELECT id FROM projects WHERE slug = $projectSlug AND clerk_org_id = $orgId
   → Not found? Return error: PROJECT_NOT_FOUND
     │
     ▼
2. SELECT * FROM feature_packs
   WHERE project_id = $projectId AND slug = $packSlug AND is_active = true
   ORDER BY version DESC LIMIT 1
   → Not found? Return error: PACK_NOT_FOUND
     │
     ▼
3. Resolve parent chain (recursive, max 10 levels):
   chain = [pack]
   current = pack
   visited = Set([pack.id])
   while current.parent_pack_id != null AND chain.length < 10:
     parent = SELECT * FROM feature_packs WHERE id = current.parent_pack_id
     if parent.id in visited: return error INHERITANCE_CYCLE
     visited.add(parent.id)
     chain.push(parent)
     current = parent
     │
     ▼
4. Merge content root → leaf:
   resolved = {}
   for pack in chain.reverse():  // root first
     for each field in pack.content:
       if field is array (tools, conventions, allowedPaths, blockedPaths):
         resolved[field] = [...(resolved[field] ?? []), ...pack.content[field]]
       else:
         resolved[field] = pack.content[field]  // child overrides parent
     │
     ▼
5. Return resolved pack with inheritanceChain metadata
```

### Tool: save_context_pack — Idempotent Save + Embedding Enqueue

```
Input: { runId, title, content, featurePackId?, metadata? }
     │
     ▼
1. Verify run belongs to org:
   SELECT project_id FROM runs WHERE id = $runId
   JOIN projects ON projects.id = runs.project_id WHERE projects.clerk_org_id = $orgId
   → Not found? Return error: RUN_NOT_FOUND
     │
     ▼
2. INSERT INTO context_packs
   (project_id, run_id, title, content, feature_pack_id, metadata, status)
   VALUES ($projectId, $runId, $title, $content, $featurePackId, $metadata, 'committed')
   ON CONFLICT (run_id, title) DO NOTHING
   RETURNING id
   → Conflict? SELECT id WHERE run_id = $runId AND title = $title (return existing)
     │
     ▼
3. Enqueue BullMQ job: queue "embed-context-pack"
   payload: { contextPackId, content: $content, projectId }
   opts: { attempts: 3, backoff: { type: "exponential", delay: 2000 } }
     │
     ▼
4. Return { id, runId, title, embeddingJobId, savedAt }
```

### Tool: search_packs_nl — Semantic Search via NL Assembly

```
Input: { projectSlug: "my-app", query: "how did we implement OAuth?", limit: 5 }
     │
     ▼
1. Resolve projectId from projectSlug + orgId
     │
     ▼
2. HTTP POST to NL Assembly internal endpoint:
   POST http://nl-assembly.internal:3200/search
   Body: { project_id: $projectId, query: $query, limit: $limit, min_similarity: 0.7 }
     │
     ▼
   NL Assembly internally:
   a. Encode query → vector[384] using sentence-transformers
   b. SELECT id, title, content, summary,
             (summary_embedding <=> $queryVec) AS distance
      FROM context_packs
      WHERE project_id = $projectId
        AND summary_embedding IS NOT NULL
        AND (summary_embedding <=> $queryVec) < (1 - $minSimilarity)
      ORDER BY summary_embedding <=> $queryVec
      LIMIT $limit
   c. Return ranked results
     │
     ▼
3. Format results for MCP response with excerpts (first 300 chars of content)
```

## 6. Database — Supabase Connection Strategy

### Connection Types

Supabase provides three ways to connect. ContextOS uses two:

| Connection | Port | Use Case | Used By |
|------------|------|----------|---------|
| **Session Pooler** (Supavisor) | 5432 | Application runtime queries | All 5 services via `DATABASE_URL` |
| **Direct** | 5432 (different host) | DDL migrations | `drizzle-kit migrate` only |
| Transaction Pooler | 6543 | Serverless functions | Not used (our services are persistent containers) |

### Why Session Pooler, Not Direct

Session pooler (Supavisor) provides:
- Connection pooling — reuses backend Postgres connections across requests.
- IPv4 support — Fly.io supports both IPv4 and IPv6, but session pooler guarantees compatibility.
- Load distribution — Supavisor balances connections to the database backend.

Direct connection is used ONLY for migrations because DDL statements (`CREATE TABLE`, `ALTER`) and `prepare` are not safe through connection poolers.

### Drizzle + postgres-js Configuration

```typescript
// packages/db/src/index.ts
import postgres from 'postgres';

// prepare: false is REQUIRED for Supabase Supavisor compatibility.
// Supabase docs: "Disable prefetch as it is not supported for Transaction pool mode"
// Safe to set unconditionally — works correctly on session mode and direct connections.
const sql = postgres(databaseUrl, { max: poolSize, prepare: false });
```

### Migration Strategy

```bash
# Local development (using docker-compose postgres):
DATABASE_URL=postgresql://contextos:contextos_dev@localhost:5432/contextos
# DATABASE_URL_MIGRATE not set → drizzle.config.ts falls back to DATABASE_URL

# Production (Supabase):
DATABASE_URL_MIGRATE=postgresql://postgres:[password]@db.[ref].supabase.co:5432/postgres
# drizzle.config.ts uses DATABASE_URL_MIGRATE for migrations
```

### pgvector on Supabase

pgvector is a built-in Supabase extension. It must be enabled once per project:
- **Dashboard:** Database → Extensions → search "vector" → Enable.
- **SQL:** `CREATE EXTENSION IF NOT EXISTS vector;` (run in the SQL editor or as the first migration).

The HNSW index in the initial migration works natively on Supabase without any special configuration.

## 8. BullMQ Job Queues

BullMQ runs **inside existing service processes** — no additional containers. Producers and workers share the same Redis connection.

### Queue Definitions

| Queue Name | Producer | Worker | Payload Schema | Purpose |
|------------|----------|--------|----------------|---------|
| `record-run-event` | hooks-bridge | hooks-bridge | `{ runId, eventType, toolName, toolInput?, toolOutput?, policyDecision?, idempotencyKey }` | Async event logging to `run_events` table |
| `context-pack-assembly` | hooks-bridge (SessionEnd handler) | hooks-bridge | `{ runId, sessionId, projectId, cwd }` | Assemble full context pack from run events |
| `embed-context-pack` | hooks-bridge (assembly worker) / mcp-server (save_context_pack tool) | nl-assembly | `{ contextPackId, content, projectId }` | Generate embedding vector and update context_packs |

### Producer/Worker Architecture

```
hooks-bridge process:
  ├── HTTP server (Hono) handles requests
  ├── BullMQ Queue: "record-run-event" (producer)
  ├── BullMQ Queue: "context-pack-assembly" (producer)
  ├── BullMQ Queue: "embed-context-pack" (producer)
  ├── BullMQ Worker: "record-run-event" (consumer)
  └── BullMQ Worker: "context-pack-assembly" (consumer)

nl-assembly process:
  ├── HTTP server (FastAPI uvicorn) handles /embed, /search
  └── BullMQ Worker: "embed-context-pack" (consumer, polls Redis)
      → Uses Python BullMQ client or Redis BRPOP-based consumer

mcp-server process:
  └── BullMQ Queue: "embed-context-pack" (producer only, on save_context_pack)
```

### Job Options

```typescript
// All queues use the same base config
const defaultJobOptions = {
  removeOnComplete: { count: 1000 },  // Keep last 1000 completed for debugging
  removeOnFail: { count: 5000 },      // Keep last 5000 failed for analysis
};

// Per-queue overrides
const queueConfigs = {
  'record-run-event': {
    attempts: 3,
    backoff: { type: 'exponential', delay: 1000 },
  },
  'context-pack-assembly': {
    attempts: 3,
    backoff: { type: 'exponential', delay: 5000 },
  },
  'embed-context-pack': {
    attempts: 3,
    backoff: { type: 'exponential', delay: 2000 },
  },
};
```

## 10. Web App UX Principles

These are **implementation constraints**, not suggestions. They apply to every page and component.

### Principle 1: Zero Wait Perception on Navigation

All dashboard pages are React Server Components. Data is fetched inline in the component body using Drizzle queries — no `useEffect`, no client-side fetch on mount. Every page uses `<Suspense>` boundaries with skeleton cards. Content streams in progressively; users never see an empty shell while data loads.

### Principle 2: Actions Feel Instant

Policy rule reorder, Feature Pack field saves, and run triggers use **optimistic UI** via React `useOptimistic`. The UI updates the moment the user acts. The server call happens in the background. On failure: state rolls back with an inline error toast (not a modal, not a full-page error).

### Principle 3: Complexity is Progressive

The Feature Pack editor shows the content form by default. Pack inheritance tree (D3 visualization), version diff view, and "test a rule" dry-runner are **secondary tabs** — visible when clicked, not occupying primary real estate. First-time users see a clean form. Power users find depth one tab away.

### Principle 4: Real-Time Where It Matters, Static Everywhere Else

Only active run cards open SSE connections (`/api/runs/[id]/events`). Completed/historical data is static RSC — no WebSocket, no polling. The SSE connection is scoped to the card component and auto-closes when the run completes. No Socket.io, no Pusher, no Ably — SSE requires zero additional infrastructure.

### Principle 5: Copy-Paste Hook Config

The project Settings page shows the `.claude/settings.json` snippet with the developer's actual token pre-filled, a "Copy" button, and nothing else. No documentation links in the UI, no multi-step instructions. Goal: paste into settings file → working in 30 seconds.

### Principle 6: Policy Rules Have a "Test" Button

Next to every policy rule row: an inline input for tool name + file path → click "Test" → badge shows which rule would match and what decision it returns. Uses the same Policy Engine algorithm (dry-run mode, no `policy_decisions` row written). Developers validate policy logic without running Claude Code.

### Principle 7: Context Pack Detail is a Three-Panel View

Left panel: metadata (date, run ID, feature pack version, file count, status badge).
Center panel: rendered Context Pack markdown content.
Right panel: Semantic Diff (APIs added/removed, tests added, breaking changes, key decisions).
All three visible simultaneously on desktop. Tabbed layout on mobile (< 768px).

## 11. Failure Modes and Recovery

### Service Failures

| Failure | Impact | Recovery |
|---------|--------|----------|
| **hooks-bridge down** | Claude Code hooks get HTTP errors. Claude Code logs error and continues (non-blocking for all hooks except PreToolUse deny). No new run events recorded. | Fly.io auto-restart. Backlog builds in BullMQ — processed when service recovers. |
| **mcp-server down** | Agents cannot fetch Feature Packs or search context. Agent LLM works without context (degraded, not broken). | Fly.io auto-restart. VS Code extension serves cached data from local SQLite. |
| **nl-assembly down** | Embedding generation stops. Semantic search returns empty results. New context packs saved but not vectorized. | Fly.io auto-restart. `embed-context-pack` jobs retry (3x exponential). Failed jobs stay in queue — processed on recovery. |
| **semantic-diff down** | Context packs saved without semantic diff analysis. `semantic_diff` column is NULL. | Fly.io auto-restart. `context-pack-assembly` jobs retry. Content still saved, just without LLM summary. |
| **web down** | Dashboard inaccessible. Does NOT affect Claude Code agents — they use mcp-server and hooks-bridge directly. | Fly.io auto-restart. |

### Database Failures

| Failure | Impact | Recovery |
|---------|--------|----------|
| **Supabase unavailable** | All services fail all DB operations. hooks-bridge defaults to `allow` on PreToolUse (fail-open). | Wait for Supabase recovery (managed). BullMQ jobs retry. Redis session cache still works for run ID lookups. |
| **Connection pool exhausted** | DB queries timeout. Same impact as unavailable. | Increase `poolSize` in `createDb()` or scale Supabase compute tier. |
| **Migration failure** | Previous schema version still active. No data loss. | Fix migration SQL, re-run. Drizzle tracks migration state in `__drizzle_migrations` table. |

### Redis Failures

| Failure | Impact | Recovery |
|---------|--------|----------|
| **Upstash unavailable** | BullMQ cannot enqueue or process jobs. Session cache unavailable (hooks-bridge falls back to DB query for run ID). Policy cache misses (falls back to DB query). | Wait for Upstash recovery. BullMQ jobs are persisted in Redis AOF — no job loss on restart. |

### Design Principle: Fail Open, Log Everything

The hooks-bridge **never blocks Claude Code** on infrastructure failure. If any dependency (DB, Redis, NL Assembly) is unavailable:
- PreToolUse → default `allow` + log error.
- All other hooks → respond `200 { continue: true }` + log error.
- Async jobs → retry via BullMQ backoff. Dead-lettered after max attempts.

### Startup Order

Services can start in any order. Each service:
1. Validates env vars at startup (Zod schema). Missing required var → crash immediately with clear error.
2. Tests DB connection (`SELECT 1`). Failure → crash (service restarts via Fly.io).
3. Tests Redis connection (`PING`). Failure → log warning, continue (graceful degradation).
4. Starts HTTP server.
5. Health endpoint (`GET /health`) reports dependency status.
